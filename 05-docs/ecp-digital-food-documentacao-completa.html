Visão Geral do Produto

Resumo executivo do FoodFlow — marketplace premium de food delivery para São Paulo.

1.1 Visão e Posicionamento

Visão do Produto
Marketplace premium de food delivery para São Paulo

Em 5 anos, FoodFlow será a plataforma de referência para quem busca experiências gastronômicas por delivery em São Paulo e nas principais capitais do Brasil — reconhecida pela curadoria impecável de restaurantes artesanais, pela experiência de descoberta que encanta, e por um ecossistema de pagamento integrado que torna pedir comida tão prazeroso quanto comer.

Seremos a ponte entre restaurantes que se importam com qualidade e consumidores que valorizam o que comem.

1.2 North Star Metric

Métrica Estrela-Guia
Pedidos Completados com Satisfação por Semana

Descrição: Número de pedidos entregues com sucesso (sem cancelamento e sem reclamação aberta) por semana ativa.

Fórmula: Σ (pedidos com status 'Entregue' e sem ticket de reclamação aberto em 48h) / semana

Meta MVP: 630 pedidos/semana (90/dia) nos primeiros 3 meses

Meta Ano 1: 6.300 pedidos/semana (900/dia)

Input Metrics

  • Taxa de Conversão (Visitor-to-Order): Meta: > 8%
  • Ticket Médio por Pedido: Meta: > R$ 45
  • Taxa de Recompra 30 dias: Meta: > 25%
  • Tempo até Primeiro Pedido: Meta: < 5 minutos
  • Restaurantes Ativos / Semana: Meta: > 40 (MVP), > 200 (ano 1)
  • NPS do Consumidor: Meta: > 40

2.2 5 Princípios do Produto

Princípio #1
Descoberta antes de tudo
Cada tela deve ajudar o consumidor a encontrar o que quer comer em menos passos. Trade-off: priorizamos experiência de descoberta mesmo que isso signifique mostrar menos restaurantes por tela.
Princípio #2
Checkout é sagrado
Do carrinho ao pedido confirmado, zero distrações e zero surpresas. Preços transparentes, cupons que funcionam, pagamento que flui. Trade-off: priorizamos velocidade e clareza no checkout mesmo que isso limite opções de upsell.
Princípio #3
Restaurante parceiro, não refém
Restaurantes devem ter controle real sobre sua presença: cardápio, preços, promoções, dados. Ranking transparente, sem 'pay-to-play' obrigatório. Trade-off: priorizamos transparência mesmo que reduza receita de mídia paga.
Princípio #4
Design como diferencial competitivo
Experiência visual memorável: dark mode premium, tipografia elegante, animações suaves, fotos que dão fome. Trade-off: investimos mais tempo em craft visual.
Princípio #5
Dados para decisões, não para dashboards
Cada métrica que coletamos deve informar uma decisão concreta. Se um dado não gera ação, não o mostramos. Trade-off: poucos indicadores acionáveis sobre painéis completos.

3.2 Insights Principais

INS-01 — Confiança alta
Gap de UX premium é o maior diferencial competitivo disponível
Todos os concorrentes priorizam volume sobre curadoria. Reviews mencionam "UI poluída" e "fadiga de decisão" como dores recorrentes.
INS-02 — Confiança alta
PIX como infraestrutura validada — mas integração inline é risco de usabilidade
150M+ usuários de PIX no Brasil. Fluxo inline (sem redirect) é crítico para conversão.
INS-03 — Confiança alta
Curadoria > volume para segmentos C1 e C2
O FoodFlow deve oferecer 50 restaurantes curados em vez de 500 genéricos.
INS-04 — Confiança alta
Checkout é sagrado — maior RICE score do portfólio
Transparência de preços desde o card do restaurante é pré-requisito.
INS-05 — Confiança alta
Repetir Pedido é o mecanismo central de retenção
RICE 37.5 com esforço baixo — melhor ratio do portfólio. Leading indicator do KR2.
INS-06 — Confiança alta
Restaurante parceiro, não refém — ranking por qualidade
Posicionamento "ranking por qualidade, não por pagamento" é a razão pela qual restaurantes escolheriam o FoodFlow.
INS-07 — Confiança média
Cold start é o risco existencial — supply-side primeiro
Estratégia: 0% comissão nos primeiros 3 meses, onboarding assistido em bairros-alvo de SP.
INS-08 — Confiança média
Design emocional Midnight Express eleva percepção de valor
Consumidores atribuem WTP 15% maior para pratos com descrição narrativa vs lista de ingredientes.

4.2 Arquitetura DDD — 6 Bounded Contexts

Identity
Autenticação, autorização, sessões e perfis
JWT 24h + Refresh Token 7d. Roles: consumer, restaurant, admin. bcryptjs 12 rounds.
Catalog
Restaurantes, categorias, cardápios e itens
Domínio público (leitura sem auth). 6 restaurantes seed, 24 itens, 7 categorias.
Ordering
Carrinho, pedidos, itens de pedido, cupons
Status: pending_payment → confirmed → preparing → out_for_delivery → delivered. Frete grátis ≥ R$ 120.
Payment
Integração ECP Digital Bank
Valores em centavos. Circuit breaker (5 falhas = 30s block). PIX expira em 10 min. HMAC-SHA256 validation.
Delivery
Tracking simulado de entregas
MVP: restaurante atualiza status manualmente. Sem GPS real. ETA calculado via eta_min/eta_max.
Admin
Dashboard e gestão da plataforma

4.3 Diagrama de Sequência — Fluxo Completo (Pedido + Pagamento)

Fluxo end-to-end: do pedido ao pagamento (Cartão ECP e PIX QR Code) com integração ao ECP Digital Bank.

Sequência de Pedido + Pagamento — Cartão ECP & PIX QR Code
Regras de Resiliência do Fluxo
  • Idempotente: mesmo payment.id reenviado não duplica transação
  • Timeout banco: 10s com 1x retry para erros 5xx
  • Circuit breaker: 5 falhas consecutivas = 30s de bloqueio
  • PIX expira: 10 minutos — após expiração, payment.status = 'expired'
  • Webhook: sempre retorna 200 OK para prevenir retries do banco
  • SSE auth: token JWT de curta duração (5 min) via query param
  • Bank JWT efêmero: nunca persistido além da transação

4.4 Architecture Decision Records (ADRs)

ADR-001
Monorepo Single-Server Architecture
Fastify serve API REST (/api/*) + React SPA (client/dist/) em um único processo Node.js. Custo mínimo (VPS R$ 20-50/mês).
ADR-002
SQLite with better-sqlite3 as Primary Database
SQLite via better-sqlite3 11.x com WAL mode. Arquivo em ./data/foodflow.db. Migrations inline no startup. Zero-config.
ADR-003
ECP Digital Bank Integration as Payment Proxy
FoodFlow atua como proxy de pagamento. bank-integration.mjs encapsula todas as chamadas. Cartão Virtual ECP + PIX QR Code com webhook + SSE.
ADR-004
SSE for Real-Time PIX Payment Notification
Server-Sent Events via GET /api/payments/:id/events. Nativo no browser (EventSource), unidirecional, baixo overhead.
ADR-005
CSS Modules with CSS Custom Properties for Theming
CSS Modules (*.module.css) para escopo isolado + CSS Custom Properties para theming global Midnight Express. Zero runtime JS.
